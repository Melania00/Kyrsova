Проектування архітектури системи керування готелем

Цей документ описує логічну структуру системи, ідентифікує ключові класи та їхні взаємозв'язки для забезпечення функціоналу бронювання, управління номерами та аналітики.

1. Ідентифікація основних сутностей

Для реалізації вимог проекту виділено наступні ключові класи:

User (Користувач): Базовий клас для автентифікації.

Client (Клієнт): Нащадок користувача, має бонусний рахунок та історію бронювань.

Admin (Адміністратор): Нащадок користувача, має доступ до звітів та управління статусами.

Room (Номер): Описує фізичний номер у готелі (категорія, ціна, місткість).

Booking (Бронювання): Сполучна сутність між клієнтом та номером на певний період.

ExtraService (Додаткова послуга): SPA, харчування, трансфер тощо.

Report (Звіт): Сутність для збору статистичних даних.

5. Опис взаємозв'язків

Успадкування (Inheritance): Клієнт та Адміністратор успадковують базові поля від User. Це дозволяє використовувати спільну таблицю в БД та єдиний механізм JWT-авторизації.

Асоціація (Association):

Клієнт може мати багато бронювань (1:*).

Номер може фігурувати у багатьох бронюваннях (1:*), але на різні дати.

Агрегація (Aggregation): Бронювання включає в себе перелік додаткових послуг через проміжну сутність BookingService. Це дозволяє гнучко розраховувати фінальну вартість проживання.

5. Схема бази даних (Логічний рівень)

Для реалізації цієї архітектури у вашій MySQL БД використовуються такі таблиці:

Users (з полем Discriminator для ролей).

Rooms (з зовнішнім ключем на категорії).

Bookings (зв'язок UserID та RoomID).

Services (довідник послуг).

BookingDetails (для збереження обраних послуг у конкретному бронюванні).

5. Дизайн інтерфейсу (Mapping)

На рівні фронтенду (React) ці сутності відображаються так:

Каталог: Відображення масиву об'єктів Room.

Кабінет: Фільтрація масиву Booking, де ClientID збігається з ID поточного користувача.

Адмін-панель: Агрегація даних з Booking та Room для розрахунку відсотка заповненості.
